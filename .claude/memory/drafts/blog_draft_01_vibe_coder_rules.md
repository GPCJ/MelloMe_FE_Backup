## 섹션 구조

### 0. 도입 — "요즘 이상한 걸 느낀다" (3~5줄)

**핵심 요지**
- AI 코딩 도구를 쓰기 시작한 뒤 생긴 위화감
- "생각하는 시간이 줄고 있다"는 체감
- 이 글에서 다룰 것: 왜 이걸 느꼈고, 어떻게 대응하고 있는가

**힌트**: 첫 문장은 작고 구체적으로.
- 예) "Cursor를 쓴 지 몇 개월쯤 됐을 때, '이 코드가 왜 이렇게 되는 거야?'라는 질문에 내가 막힌 순간이 왔다."

---

### 1. '바이브 코더'란 무엇인가 + '인지부채' 개념

**핵심 요지**
- **바이브 코더(Vibe Coder)** 정의 — AI 제안을 '감(vibe)'으로 따라가며 코딩하는 개발자 (Andrej Karpathy 원조 용어)
- **인지부채(Cognitive Debt)** — AI에 생각을 위임하면 당장은 편하지만, 스스로 사고하는 능력이 약해지는 현상
- 근거 연구 (⚠️ **직접 검색해서 원문 확인 후 인용**):
  - **MIT Media Lab 2025** — *"Your Brain on ChatGPT: Accumulation of Cognitive Debt..."* ← 인지부채 용어 실제 출처
  - 검색: `MIT Your Brain on ChatGPT cognitive debt`
  - (선택) GitClear 2024 보고서 — AI 사용 시 코드 품질/중복 증가
  - 검색: `GitClear 2024 AI code quality report`

**힌트**:
- 용어 설명은 **2~3줄 내외**로 짧게. 장황해지면 이탈.
- 인용할 거면 반드시 원문 확인 후. "MIT 연구에 따르면..."처럼 뭉뚱그리지 말고 **연도/연구명/핵심 결론** 구체적으로.

---

### 2. 내가 체감한 순간들 (일화 1~2개) ⭐ 글의 심장

**핵심 요지**
- 추상적 "사고력이 떨어지는 것 같다"는 공감 안 됨 → **구체적 일화 필수**
- 예시 후보 (본인 실제 경험에서 상세히 1~2개 고르기):
  - AI가 짜준 훅을 push하고 다음날 다시 보니 내가 쓴 게 아닌 것처럼 낯설었던 순간
  - 디버깅 중 AI에 답만 구하다가 "내가 이 문제 왜 일어났는지 모르고 있다"는 자각
  - 코드 리뷰받을 때 "이거 왜 이렇게 했어?"에 답 못 한 경험
  - 과거엔 쉽게 풀던 타입/비동기 문제가 AI 없이 막막해진 순간

**힌트**:
- **창피한 경험을 솔직히 쓰는 게 글의 힘**. 감추면 공감 못 받음.
- 1개 깊게 vs 2~3개 짧게 — 본인 판단. 길이 여유 있으면 1개 메인 + 간단 언급 2개 구조도 좋음.

### 3. 내가 실천하는 규칙 5가지

**각 규칙마다 템플릿**:
```
### 규칙 N. [한 줄 요약]

왜 필요한가 (2~3줄)
어떻게 실천하나 (2~4줄, 구체적 행동)
효과 (1~2줄, 선택)
```

---

#### 규칙 1. 코드는 직접 쓴다 — AI는 힌트 제공자까지만

**왜 필요한가**
- AI가 타이핑한 코드는 손/머리에 남지 않음
- "이해했다"는 착각만 남고 체화는 없음

**어떻게 실천하나**
- AI에겐 방향/개념/예시만 요청
- 타이핑은 반드시 직접. 복붙 금지, 참고해서 다시 작성
- "코드 대신 설명해줘"로 요청 전환

**효과 (본인 체감한 걸로 교체)**
- 예: 같은 패턴 만나면 AI 없이 쓸 수 있는 비율 증가

---

#### 규칙 2. '해줘' 대신 '왜 이런지 진단해줘'부터 묻는다

**왜 필요한가**
- "해줘"는 생각 외주. AI 답 그대로 받고 끝
- "진단해줘"는 본인이 이해해야 검증 가능 → 사고력 살아있음

**어떻게 실천하나**
- 순서: **진단 → 이해 → 지시 → 조치**
- 버그 만나면 "고쳐줘" ❌ → "원인 후보 3개 + 각 근거 알려줘" ✅
- AI 답 받아도 **납득 못 하면 조치 안 함** (답이 맞아 보여도)

**효과 (본인 체감)**
- 예: 이해 못 하고 넘어가는 사례 줄어듦, 해결 속도는 약간 느려지지만 재발 확률 훨씬 낮음

---

#### 규칙 3. AI 답변을 맹신하지 않고 직접 검증한다

**왜 필요한가**
- AI는 hallucinate — 없는 API, 없는 패키지, 틀린 스펙 자신 있게 말함
- AI 말이 내 프로젝트 실제 상태와 다를 수 있음

**어떻게 실천하나**
- AI가 "파일 X에 Y가 있다" → 실제 파일 열어 확인
- AI가 "라이브러리에 Z 메서드 있다" → 공식 문서/타입 정의 확인
- **grep/find는 AI보다 내 손이 먼저**

**효과 (본인 체감)**
- 예: "AI가 그러던데요"라고 변명할 일 없음, 코드리뷰에서 "왜?" 질문에 막힘 없이 답변

---

#### 규칙 4. AI가 쓴 코드엔 '왜' 주석을 달게 해서 학습한다

**왜 필요한가**
- 급할 때 맡긴 코드는 나중에 내가 못 읽음
- 주석 없이 넘어가면 인지부채가 그냥 쌓임

**어떻게 실천하나**
- push 전에 AI에게 **"교육용 밀도 높은 한국어 주석 추가해줘"** 요청
- 주석을 **내가 읽고 이해한 뒤에만** 승인
- 이해 안 되는 주석은 다시 물어 명확해질 때까지 진행 안 함

**효과 (본인 체감)**
- 예: 주석이 나중의 내 학습 노트가 됨, rubber duck 효과로 AI 답변 품질도 올라감

---

#### 규칙 5. AI에 맡긴 부분은 '인지부채'로 플래그하고 복습한다

**왜 필요한가**
- 모든 코드를 직접 쓸 수는 없음 (시간/난이도 현실 제약)
- 맡긴 부분을 기억 못 하면 나중에 "블랙박스"가 됨

**어떻게 실천하나**
- AI 위임 코드 영역을 별도 기록 (메모리/노트)
- 메커니즘 상세 문서화 (단순 "fix됨"이 아니라 "왜/어떻게")
- 주기적으로 꺼내서 분해 학습 (읽기만 말고 지웠다 다시 짜보기)

**효과 (본인 체감)**
- 예: 블랙박스가 줄어듦, "내 코드베이스"라는 감각 유지

---

### 4. 규칙 너머 — AI가 내 정보를 어떻게 쓰는지 아는 것 ⭐ 글의 차별점

**핵심 요지**
- 5가지 규칙은 **"코드를 대하는 자세"**
- 그 위에 한 층 더 중요한 게 있음 → **AI 시스템 자체를 이해하는 것**

**구체적으로 알아야 할 것**
- AI가 내 메모리/히스토리를 **어떻게 참조**하는지 (컨텍스트 윈도우, 시스템 프롬프트, 메모리 타입별 취급)
- 어떤 정보를 저장/참조하고 어떤 건 잊는지
- 프롬프트를 어떻게 해석하는지 (instruction following, hallucination 패턴)
- 내 설정/`CLAUDE.md`/메모리가 AI 응답에 어떻게 반영되는지

**왜 중요한가**
- 모르면 → "왜 이런 답이 나왔지?" 이해 못 하고 그냥 받아들임
- 알면 → AI를 **예측 가능한 도구**로 쓸 수 있음. 엉뚱한 답 나올 때 원인 추적 가능
- 비유: **자동차를 모는 사람 vs 엔진 구조를 아는 사람** — 안전성/제어력 차이

**구체 예시 (본인 경험에서 1~2개 골라 상세히)**
- Claude Code 메모리 타입(project/feedback/user/reference) 구조 파악 후 의도적으로 분류해 저장
- 컨텍스트 윈도우 한계 알고 중요 정보 우선 배치
- `CLAUDE.md`/시스템 프롬프트 규칙이 AI 응답에 어떻게 영향 주는지 관찰하고 조정
- 메모리 인덱스(`MEMORY.md`)가 자동 로드된다는 점 활용해 핵심 정보 배치

**힌트**:
- 이 섹션이 **글의 차별점**. 대부분의 "AI 시대 규칙" 글은 코드만 다룸 → 여기서 한 층 더 올라가는 게 본인의 특색
- 구체 예시가 핵심. 추상적 "AI 원리도 알아야 한다" 뿐이면 공허함

### 5. 맺음 — 앞으로의 다짐 / 독자에게

**핵심 요지**
- AI를 버리는 게 답이 아님. 잘 쓰되 **주체는 나**라는 원칙
- 5가지 규칙은 완성된 답이 아니라 **진행형 실험**. 계속 다듬는 중
- **코드 원리도 중요하지만 AI 원리도 중요하다** — 4번 섹션 메시지 한 번 더 강조
- 독자 초대: 당신의 규칙은? 어떤 순간 인지부채를 체감하나?

**힌트**:
- 5~8줄, 담백하게
- 설교형 금지. "너도 이렇게 해야 해" ❌ / "나는 이렇게 하고 있다" ✅
