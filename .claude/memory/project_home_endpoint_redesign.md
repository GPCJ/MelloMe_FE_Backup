## 결정 (2026-04-04)

LandingPage 렌더링 시 매번 `/me`로 과도한 데이터를 받아오는 문제를 jin24님이 직접 발견.
`/home` 엔드포인트가 이미 존재하지만 응답이 비어있는 상황에서, 관심사 분리 관점으로 역할을 재정의함.

## 핵심 설계

- **`/home`** → 랜딩페이지 전용 경량 데이터
- **`/me`** → 마이페이지/프로필 등 전체 유저 정보

## MVP 응답 스펙 (백엔드 요청 필요)

```json
{
  "nickname": "string",
  "canAccessCommunity": boolean,
  "verificationStatus": "APPROVED" | "PENDING" | "REJECTED" | "NOT_REQUESTED"
}
```

## 확장 방향

- 추후 랜딩페이지에서만 필요한 데이터(공지사항, 인기 게시글 등)를 `/home`에 추가
- 알림은 SSE 백엔드 준비 후 추가

## 역량 어필 포인트

- **문제 발견**: 렌더링마다 /me 호출로 불필요한 데이터 수신하는 비효율을 스스로 인지
- **설계 판단**: 기존 /home 엔드포인트를 활용한 관심사 분리 제안 — 새 엔드포인트 생성 없이 기존 자원 재활용
- **확장성 고려**: 단순 최적화가 아닌, 향후 데이터 추가를 고려한 엔드포인트 역할 분리 설계
