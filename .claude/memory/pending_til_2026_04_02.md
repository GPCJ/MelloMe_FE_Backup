## 2026-04-02 — SSE 알림 기능 아키텍처 설계 (3단계 의사결정)

**분류**: React / 보안

### 오늘 한 것
- SSE 기반 실시간 알림 기능의 프론트엔드 아키텍처 설계 완료 (3단계)
  - **0단계 — 실시간 기술 선택**: Polling(불필요한 반복 요청) / WebSocket(양방향이라 과함) / SSE(단방향 push에 적합) 비교 → SSE 채택
  - **1단계 — 프론트 상태 관리 구조**: 3가지 옵션 비교 후 옵션 B(Zustand store에 SSE 연결 + 알림 상태 통합) 확정
    - A. useSSE 훅 + store 분리 → 컴포넌트 언마운트 시 연결 끊김 위험
    - B. Zustand store 통합 → 연결 생명주기가 컴포넌트와 독립, 기존 패턴 일관성 ✅
    - C. 싱글톤 클래스 + store → MVP 규모에 과한 추상화
  - **2단계 — SSE 토큰 인증 방식**: 브라우저 기본 EventSource API가 Authorization 헤더를 지원하지 않는 문제 → `@microsoft/fetch-event-source` 라이브러리로 해결
    - 쿼리 파라미터(`/sse?token=xxx`)는 URL에 토큰 노출 → 보안 우려
    - fetch 직접 파싱은 재연결/에러처리 구현 부담
    - 라이브러리가 내부적으로 fetch 기반 SSE 연결을 열어 헤더 커스터마이징 지원

### 배운 것 / 인사이트
- **SSE vs 일반 HTTP**: 일반 API는 요청-응답 후 연결 종료, SSE는 최초 연결 후 서버가 계속 이벤트를 push하는 구조. 알림 클릭 → 상세 조회는 별도 일반 API 호출이면 충분
- **EventSource의 설계 의도**: 공개 데이터 스트리밍(주식 시세, 뉴스 등)용으로 만들어져 헤더 커스터마이징이 애초에 필요 없었음. "하자"가 아니라 설계 목적이 다른 것
- **Zustand store의 생명주기 이점**: React 훅은 컴포넌트에 묶이지만 Zustand store는 컴포넌트 밖에 존재 → 페이지 이동해도 SSE 연결이 유지되는 구조적 장점

### 포트폴리오 어필 포인트
- 🏗 **체계적 아키텍처 설계**: 실시간 기술 선택 → 상태 관리 구조 → 인증 방식까지 3단계에 걸쳐 각각 복수 옵션의 트레이드오프를 분석하고 프로젝트 규모에 맞는 선택을 도출
- 🤖 **AI 협업 설계 프로세스**: AI 브레인스토밍으로 옵션 도출 → 각 옵션의 장단점을 직접 이해하고 질문 → 최종 의사결정은 본인이 수행
- 📈 **확장성 고려**: MVP에서는 옵션 B로 시작하되, 알림 종류 증가 시 옵션 C(서비스 클래스)로 마이그레이션 가능한 경로 확보
