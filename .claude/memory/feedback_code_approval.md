코드를 생성하거나 파일을 작성/수정하기 전에 승인을 요청할 때는, 단순히 "이 파일을 만들겠습니다"가 아니라 다음 내용을 포함해서 설명해야 한다:

- 어떤 파일을 만들거나 수정하는지
- 그 파일이 어떤 역할을 하는지
- 주요 로직이나 구조가 어떻게 되는지
- 왜 이렇게 설계했는지 (필요한 경우)

예시:
> `src/api/auth.ts`를 생성할게요. 이 파일은 `/auth/login`, `/auth/oauth/google` 엔드포인트를 호출하는 함수를 담습니다. 에러 처리는 `handleResponse` 헬퍼로 통일하고, 실패 시 `Error`를 throw해서 호출부에서 catch하도록 했어요.

## 단계별 승인 예외 (2026-04-05 추가)
순차적으로 여러 항목을 고칠 때는 **단계마다 승인 묻지 말고 바로 다음 단계 진행**.
사용자가 "하나씩 진행해줘"라고 한 경우, 각 단계 완료 후 결과만 보고하고 다음으로 넘어가면 됨.
승인은 전체 작업 시작 전 방향 합의 시에만 받는다.

## 설계 승인 ≠ 코드 작성 승인 (2026-04-22 추가)
"A안으로 진행" 같은 **방향/설계 합의**는 곧바로 코드 수정을 의미하지 않는다. 사용자는 `user_self_coding_goal.md` 원칙에 따라 **직접 코딩**을 기본값으로 삼는다.

**How to apply:**
- 방향 합의 직후 + 코드 변경 도구(Edit/Write) 호출 직전 사이에 **"직접 작성하시겠어요, 아니면 제가 작성할까요?"** 한 번 확인
- 기본값 가정 금지 (자동으로 AI가 쓰는 것도 ❌, 자동으로 사용자가 쓴다고 단정하는 것도 ❌)
- 사용자가 그 세션에서 명시적으로 "네가 써줘"라고 한 경우에만 재질문 스킵 가능

**계기:** 2026-04-22 R-01 RQ 마이그레이션 A안 승인 후 AI가 곧바로 Edit 호출 → 사용자 거부 "너가 코드 수정할거야?"
