결정/구현 문서를 작성할 때 강점·근거뿐 아니라 **"현재 조치의 한계점"** 섹션을 별도로 두어 함께 박제합니다.

**Why:**
- 미래 재검토 시점(예: "게시글 공개 정책 변경 시 Next.js 재평가")에 한계 기록 자체가 추적 단서가 됩니다. 한계 없이 강점만 남으면 재평가 트리거를 놓칩니다.
- 의사결정 과정에서 "이건 noindex 권장이라 OK", "MVP 규모에선 무관" 같은 **합리화로 가린 부수 효과**(예: 카톡 공유 시 미리보기 빈 카드, 페이지 수 확장 시 빌드 시간 선형 증가)를 박제하지 않으면 같은 함정에 다시 빠집니다.
- "기능적 동작 = 도메인 효용"으로 착각하는 격차도 한계 섹션이 잡아냅니다. 빌드 통과·preview 응답 확인까지 했어도 실제 봇 동작(Search Console 색인, 카톡 미리보기, metatags.io 검증)은 별도 검증이 남습니다.

**How to apply:**
- 노션 결정문/회고/구현 보고 작성 시 강점/근거 섹션 다음에 `## 현재 조치의 한계점` 또는 동등 섹션을 추가합니다.
- 한계는 번호 매겨 박제하고, 각 한계마다 "왜 지금은 괜찮은가"와 "언제 문제가 되는가"를 함께 적습니다.
- 강점만 적힌 초안은 자기검열을 의심하고 다시 보며 "한계가 없는 결정은 없다"는 전제로 점검합니다.
- "검증되지 않은 항목"은 별도 박제 — "기능적으로 동작한다"와 "도메인 효용을 달성한다"는 다른 명제임을 명시합니다.

**사례 (2026-04-27 SEO 옵션 2):**
- `notion_draft.md`에 8가지 한계점 섹션 추가: 정적 페이지에만 작동, prerender 외 라우트는 SPA, 빌드 시점 메타 박제, 페이지 수 확장 시 빌드 시간 증가, JS 실행 결과 미포함, SSR-safe 가드의 우연한 import 그래프 의존, 도구 메인테이너 1인 리스크, 봇 실측 미완.
- 한계점 박제로 "게시글 공개 정책 전환 시 재평가" 트리거가 단서를 갖게 됨.
