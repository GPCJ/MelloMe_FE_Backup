메모리에 저장된 규칙(예: "push-mello 전에 pull-mello 실행")을 그대로 인용하거나 사용자에게 제약으로 전달하지 말고, 실제 실패 원인을 코드/스크립트에서 먼저 파악한다. 필요하면 워크어라운드가 아닌 근본 수정(가드 추가 등)을 제안한다.

**Why:** 2026-04-20 push-mello 실행 직전, 기록된 "pull-mello 먼저" 규칙을 근거로 사용자에게 "pull 먼저 돌릴까요?"를 제안했음. 사용자가 "이 제한사항 뭔가 잘못됐는데? 그 전례의 원인을 모르면서 그냥 막는거야?"라고 지적. 실제 원인을 스크립트에서 확인해보니 `rsync --delete`의 빈 소스 문제였고, 올바른 해결은 "pull 먼저 실행"이 아니라 스크립트에 `src_count < 5` 빈 소스 가드 추가였음. 규칙 암기 → 워크어라운드 전달 패턴이 피드백의 대상.

**How to apply**
- 메모리 규칙 적용 전: 관련 스크립트/코드를 읽어 실제 실패 메커니즘 확인.
- 워크어라운드 전달하기 전에 근본 수정 가능 여부 검토 → 가능하면 수정 제안.
- 사용자가 규칙에 회의적인 질문을 하면 맹목 방어하지 말고 전례 조사 → 원인 설명 → 수정안 제시 순서로 대응.
