## 서브 페이지: 커밋 히스토리가 왜 지저분한가요?

**Q. 커밋 히스토리를 보면 자동 커밋이 많은데, 왜 이렇게 관리했나요?**

MVP 개발 초기에는 기능 구현 속도를 최우선으로 두고 빠른 프로토타이핑에 집중했습니다. 두 개발 환경(맥북/WSL2)을 오가며 작업하다 보니, 환경 간 코드 동기화를 위한 자동 백업 스크립트를 사용했고 그 과정에서 동기화용 커밋이 다수 생성되었습니다.

핵심 기능이 어느 정도 완성된 시점에서 커밋 히스토리의 가독성 문제를 인지했고, 이후부터는 기능 단위 커밋 + conventional commit 컨벤션으로 전환했습니다.

**핵심:** 몰라서가 아니라 의식적으로 속도에 우선순위를 둔 결과이며, 문제를 인지한 뒤 스스로 개선했습니다.

# 트러블슈팅 — 새 페이지 (2026-04-09)

**위치:** 트러블슈팅 페이지 하위 서브 페이지로 추가

## 서브 페이지: 모바일 하단 고정 입력창 — Layout 내부 h-dvh 충돌

### 상황

댓글 시스템 UI를 3페이지 구조로 리디자인한 뒤, 코드 리뷰 과정에서 두 가지 레이아웃 문제를 발견했다.

### 문제 1: CommentWritePage 하단 입력창이 화면 밖으로 밀림

댓글 작성 페이지에서 하단 입력창을 화면 바닥에 고정하기 위해 `h-dvh`(= `height: 100dvh`) + `flex-col` 구조를 사용했다.

```
<div class="h-dvh flex flex-col">     ← 뷰포트 전체 높이
  <div class="flex-1 overflow-y-auto"> ← 스크롤 영역
    게시글 본문
  </div>
  <div>댓글 입력창</div>              ← flex 바닥에 붙음
</div>
```

의도대로라면 입력창이 화면 바닥에 위치해야 했지만, **실제로는 화면 아래로 밀려서 보이지 않았다.**

### 원인 분석

이 페이지는 단독으로 렌더링되지 않는다. Layout 컴포넌트의 `<main>` 안에서 렌더링된다.

```
Layout
├── <header>              ← 높이 차지
├── <main class="flex-1"> ← 이 안에서 페이지 렌더링
│   └── CommentWritePage
│       └── h-dvh (100dvh) ← 뷰포트 전체 높이를 요구
└── <nav>                 ← 하단 nav 높이 차지
```

`h-dvh`는 뷰포트 전체 높이(100dvh)를 차지하는데, header와 하단 nav가 이미 공간을 사용하고 있다. 결과적으로 `header 높이 + 100dvh + nav 높이` = **화면보다 커서** 하단 입력창이 밀려나는 현상이 발생했다.

핵심: **컴포넌트가 렌더링되는 부모 레이아웃 구조를 고려하지 않고 뷰포트 기준 높이를 사용한 것**이 근본 원인이었다.

### 문제 2: CommentDetailPage 불필요한 하단 여백

대댓글 페이지에서 하단 답글 입력창 영역 확보를 위해 `pb-24`(96px)를 항상 적용하고 있었다. 그런데 답글 입력창은 💬 버튼 클릭 시에만 나타난다. 입력창이 없을 때도 96px 빈 공간이 존재하는 불필요한 여백 문제.

### 해결: MobileFixedBottom 컴포넌트

두 문제의 공통 원인은 **하단 입력창을 문서 흐름(flex 레이아웃) 안에 배치한 것**이었다. `position: fixed`로 문서 흐름에서 분리하면 두 문제 모두 해결된다.

**useKeyboardHeight 훅**
```
window.visualViewport API로 모바일 가상 키보드 높이를 실시간 추적.
innerHeight - visualViewport.height - offsetTop = 키보드 높이
```

**MobileFixedBottom 컴포넌트**
```
position: fixed + bottom: keyboardHeight
→ 키보드가 없으면 화면 바닥, 키보드가 있으면 키보드 바로 위에 위치
```

적용 후:
- **문제 1 해소**: 입력창이 fixed로 문서 흐름에서 빠졌으므로 h-dvh + flex 구조 자체가 불필요해짐. 제거하고 플랫 구조로 변경.
- **문제 2 해소**: `pb-24`를 입력창 표시 여부에 따라 조건부 적용 (`showReplyInput ? 'pb-24' : 'pb-6'`).
- **보너스**: 모바일 키보드가 올라와도 입력창이 키보드 위에 정확히 위치하는 UX 개선까지 달성.

### 배운 점

1. **컴포넌트는 고립적으로 동작하지 않는다.** 부모 레이아웃(Layout → main → Outlet)까지 포함한 전체 렌더링 트리를 고려해야 한다. `h-dvh`가 단독 페이지에서는 정상 동작하지만, Layout 안에서는 깨진다.
2. **하나의 근본 원인을 해결하면 여러 증상이 동시에 사라진다.** "문서 흐름 안에서 하단 고정"이라는 공통 원인을 `position: fixed`로 해결하니 두 문제 + 키보드 대응까지 한 번에 처리됐다.
3. **코드 리뷰가 있어야 문제가 보인다.** 기능 구현에 집중할 때는 "동작하니까 OK"로 넘어가지만, 리뷰 단계에서 구조적 결함을 발견할 수 있었다.
