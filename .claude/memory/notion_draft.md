# #005 — 탈퇴 계정 로그인 시 무한 로딩 (axios 인터셉터 데드락)

**날짜**: 2026-04-06
**분류**: 인증 / axios 인터셉터
**난이도**: ⭐⭐⭐

## 문제 상황
소프트 삭제(탈퇴) 처리된 계정으로 로그인을 시도하면 "로그인 중..." 상태가 영원히 풀리지 않는 버그가 발생했다. 에러 메시지도 표시되지 않고, 버튼도 비활성 상태로 멈췄다.

## 원인 분석
axios 응답 인터셉터에서 401 에러 발생 시 자동으로 refresh 토큰을 사용해 access token을 재발급받는 로직이 있다. 문제는 이 refresh 요청 자체도 같은 axios 인스턴스를 사용하기 때문에, refresh가 401로 실패하면 동일한 인터셉터를 다시 탄다는 것이다.

**데드락 발생 흐름:**
1. 로그인 요청 → 서버 401 응답
2. 인터셉터가 401을 감지 → `isRefreshing = true` → refresh 요청 발송
3. refresh도 401로 실패 → 인터셉터가 다시 401을 감지
4. `isRefreshing`이 이미 `true`이므로 refresh 요청이 대기 큐(`failedQueue`)에 들어감
5. 큐를 해소하는 `processQueue()`는 refresh의 `await`가 끝나야 호출됨
6. 그런데 그 `await`에 걸린 Promise가 바로 큐에서 대기 중인 요청 → **순환 의존으로 영원히 pending**

`finally` 블록에 도달할 수 없으므로 `setLoading(false)`가 실행되지 않아 무한 로딩이 발생한 것이다.

## 해결 과정
refresh 요청에 `_retry: true` 플래그를 붙여서, refresh의 401 응답이 인터셉터의 토큰 갱신 로직을 타지 않고 바로 `Promise.reject`되도록 수정했다. 이렇게 하면 `catch` → `clearAuth()` → `finally` → `setLoading(false)` 순서로 정상 진행된다.

장기적으로는 refresh 전용 axios 인스턴스를 분리하는 것이 더 깔끔한 구조이므로, 리팩토링 시점에 전환 예정이다.

## 핵심 개념
- **데드락(Deadlock)**: A가 끝나야 B가 실행되고, B가 끝나야 A가 실행되는 순환 의존 상태. 이 경우 `processQueue()` 호출 ↔ refresh Promise 해소가 서로를 기다렸다.
- **axios 인터셉터의 재귀성**: axiosInstance 내부에서 axiosInstance를 사용하면 그 요청도 동일한 인터셉터를 탄다. 어떤 요청이 인터셉터를 타야 하고 어떤 요청은 타면 안 되는지 구분이 필요하다.

## 면접 포인트
- Q: "axios 인터셉터에서 토큰 갱신 시 주의할 점은?"
- A: refresh 요청이 같은 인터셉터를 타면 데드락이 발생할 수 있다. refresh 요청은 토큰 갱신 로직을 우회하도록 플래그를 붙이거나, 별도 axios 인스턴스를 사용해야 한다.

## 설계 결정: 검색 기능 코드 리뷰 → 컴포넌트 구조 개선 전략 수립 (04-07)

**대상 페이지:** 🏗 설계 결정 & 아키텍처
**URL:** https://www.notion.so/32dc8200749b81e899bde7aea0a37937

### 배경: 왜 구조적 개선이 필요하다고 느꼈는가

검색 기능(SearchPage)의 코드를 직접 읽으며 리뷰한 결과, 하나의 파일(240줄)에 검색 로직, 필터 UI, 결과 목록, 페이지네이션이 모두 섞여 있었다. useState가 8개에 달했고, 그 중 `searched`는 `results === null`로 대체 가능한 중복 상태였다. 또한 `therapyArea`, `sortType`, `currentPage`가 URL과 동기화되지 않아 새로고침 시 필터가 초기화되는 버그도 발견했다.

이런 문제들이 "기능은 동작하지만 수정하기 어려운 코드"라는 신호라고 판단했다.

### 분석: 어떤 개선이 가능한지

코드를 기능 단위로 분해했을 때, 다음과 같이 분리할 수 있었다:

- **커스텀 훅 (`useSearch`)** — 검색 로직(API 호출, 상태관리, useEffect)을 페이지에서 분리
- **공통 컴포넌트** — `Pagination`(재사용 가치 높음), `FilterChips` 등
- **상수 분리** — 필터/정렬 옵션을 별도 파일로

### 의사결정: 왜 전면 리팩토링을 선택하지 않았는가

검색 페이지만 깔끔하게 분리하면 다른 페이지와 구조적 일관성이 깨진다는 문제를 인식했다. 그렇다고 프로젝트 전체를 한 번에 리팩토링하면:

- 수정 범위가 넓어 버그 발생 확률이 높고
- MVP 출시 일정에 영향을 주며
- 한 번의 대규모 변경은 롤백도 어렵다

이 트레이드오프를 고려해 4가지 방안을 비교 검토했다:

| 방안 | 핵심 | 채택 여부 |
|------|------|-----------|
| A. 새 코드부터 적용 | 기존 코드는 건드릴 때만 정리 | ✅ 병행 채택 |
| B. 한 페이지씩 순차 리팩토링 | 페이지 단위로 커밋 | MVP 이후 고려 |
| C. 공통 컴포넌트만 먼저 추출 | 재사용 가능한 것만 분리 | ✅ 우선 채택 |
| D. 컨벤션 문서 먼저 | 팀 합의 후 코드 수정 | MVP 단계에서는 과투자 |

### 결론: 선택한 전략과 실행 계획

**C + A 조합 전략**을 채택했다.

1. **즉시 실행** — `Pagination` 컴포넌트를 공통 컴포넌트로 추출 (SearchPage, PostListPage 등에서 재사용)
2. **컨벤션 설정** — "앞으로 새 페이지는 훅/컴포넌트 분리 구조를 따른다" 규칙 추가
3. **MVP 이후** — 기존 페이지들을 수정할 일이 생길 때 같이 구조 정리

### 이 과정에서 보인 역량

- **코드 리뷰**: 동작하는 코드에서 중복 상태, URL 미동기화, race condition 등 8개 이슈를 스스로 발견
- **의사결정**: "더 좋은 코드"와 "현실적 제약(일정, 리스크, 팀 일관성)" 사이에서 근거 있는 선택
- **팀 관점**: 개인의 코드 품질이 아닌 프로젝트 전체의 일관성을 먼저 고려
