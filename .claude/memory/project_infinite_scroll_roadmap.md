## 배경

- 현재 offset 기반 페이지네이션 API 이미 동작 중
- 백엔드 개발자가 난이도를 높게 평가했으나, 멜로미 규모에서는 초중급 수준
- **MVP 단계에서는 백엔드 변경 없이 프론트만으로 구현 가능**

## MVP — "일단 동작하는 무한 스크롤"

- `IntersectionObserver`로 리스트 하단 감지
- 기존 `page` 파라미터 +1 해서 API 호출
- 응답 데이터를 기존 리스트에 append
- `isLast`/`last` 값으로 마지막 페이지면 호출 중단
- 로딩 스피너 표시
- **BE 변경: 없음** / 작업량: 1~2일

## MVP+ — "쾌적한 무한 스크롤"

- 낙관적 프리페칭 (리스트 70% 지점에서 다음 페이지 호출)
- 스크롤 위치 복원 (게시글 클릭 → 뒤로가기 시 원래 위치)
- 빈 상태 / 에러 처리 UI ("게시글 없음", "재시도")
- Skeleton UI (스피너 대신 placeholder)
- **BE 변경: 없거나 미미** / 작업량: 추가 2~3일

## 언젠가 — "대규모 서비스급"

- Cursor 기반 페이지네이션 (새 글 올라와도 중복/누락 없음, BE 변경 필요)
- Virtualization (`react-virtuoso` 등) — DOM에 보이는 항목만 렌더링
- 새 글 알림 토스트 ("새 게시글 N개")
- React Query 도입 (캐싱, 백그라운드 리페칭, stale 관리)
- **BE 변경: cursor API 필요** / 작업량: 설계 포함 1~2주

## 한눈에 비교

| | MVP | MVP+ | 언젠가 |
|---|---|---|---|
| 페이지네이션 | offset (현재) | offset (현재) | cursor (BE 변경) |
| 스크롤 감지 | 바닥 도달 | 70% 프리페칭 | + virtualization |
| UX | 스피너 | skeleton + 위치복원 | + 새 글 알림 |
| BE 변경 | 없음 | 없음 | 있음 |

## 팀 공유 시 포인트

- "현재 페이지네이션 API 그대로 쓰면 되지 않나?" 로 가볍게 확인
- 열린 질문보다 기준점이 있는 질문이 효과적
- 예: "1차로는 기존 API 활용해서 프론트에서 스크롤 감지만 추가하려고 해요. 추가 고려사항 있으면 말해주세요."
