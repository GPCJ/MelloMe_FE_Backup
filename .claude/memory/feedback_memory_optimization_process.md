## 문제
04-06 최적화에서 3개 에이전트가 100+개 파일을 전수 조사 → 10K+ 토큰 소비. 메모리 최적화는 비코딩 작업인데 과잉 비용.

## 토큰 낭비 원인
1. **에이전트 3개 병렬 투입** — 각각 30~40개 파일을 전부 Read (가장 큰 비용)
2. **통합 대상 파일도 전부 Read** — 흡수처 + 원본 양쪽 모두 읽음
3. **MEMORY.md 전체 재작성** — 부분 편집 대신 Write로 전체 교체

## 다음 최적화 시 절약형 프로세스

### Phase 1: 인덱스만으로 1차 분류 (에이전트 없이)
- MEMORY.md만 읽고 인덱스 설명(한 줄 hook)으로 삭제/통합/유지 판단
- ✅ 표시 항목 → 무조건 삭제 후보
- 날짜가 2주+ 지난 "대기/진행 중" → 상태 확인 후보
- 1줄 hook만으로 판단 불가한 것만 파일 Read

### Phase 2: 불확실한 것만 선택적 Read
- Phase 1에서 "읽어봐야 판단 가능"인 것만 Read (10개 이하 목표)
- 에이전트 투입 금지 — 직접 Read/Grep으로 처리

### Phase 3: 일괄 실행
- 삭제: `rm -f` 한 번에 처리
- 통합: 흡수처만 Read → Edit 한 번 → 원본 rm
- MEMORY.md: Edit으로 부분 수정 (전체 재작성 지양)

### 예상 효과
- 에이전트 0개 → 토큰 60~70% 절감
- 파일 Read 10개 이하 → 추가 30% 절감
- 목표: 메모리 최적화 1회당 3K 토큰 이하

**Why:** 메모리 최적화는 정기적으로 해야 하는 유지보수 작업. 매번 10K+ 쓰면 비효율.

**How to apply:** 사용자가 메모리 최적화 요청 시 이 프로세스대로 진행. 에이전트 투입은 50개+ 신규 파일이 있을 때만.
