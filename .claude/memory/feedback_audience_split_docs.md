PM(비개발자)과 본인(개발자)처럼 **이원 독자**가 예상되는 의사결정 문서는 처음부터 **2개 파일로 분리**해서 작성한다. "상단은 PM용 요약, 중/하단은 개발자용" 하이브리드 구조는 두 독자 모두에게 불친절하다.

**각 버전별 원칙**

- **PM용 (`<topic>-pm.md`)**
  - 코드 블록 전면 제거
  - 기술 용어는 첫 등장 시 inline 풀어쓰기 (예: `client_id` = "브라우저에 저장되는 쿠키 값")
  - 오너십/다음 액션 섹션을 **문서 맨 앞**으로 이동 → PM이 첫 문단에서 "다음에 뭘 해야 하는지"를 파악
  - 표는 최소 1개 (핵심만), 표 뒤에 서사 1~2문장 추가로 의미 부여
  - 목표 분량: 원본의 40%

- **개발자용 (`<topic>-dev.md`)**
  - 코드 블록·훅 디테일 유지
  - `feedback_ai_prose_patterns.md` 체크리스트 적용 (em-dash·3의법칙·볼드라벨·h3과밀 제거)
  - 실질 2분기로 선택지 축소 (허수 옵션 버리기)

**Why:** 2026-04-24 ga4-clarity-4.md 1차 문서에서 "상단 PM용 + 하단 본인용" 하이브리드 구조를 시도했으나 실패. PM은 기술 노이즈에 막혀 읽지 못하고, 본인은 상단 요약 중복으로 낭비. 분리 후 ga4-clarity-pm.md(75줄)와 ga4-clarity-dev.md(124줄)로 자리 잡음. 원본 대비 PM 68% 감량, 개발자 47% 감량.

**How to apply:**
- 의사결정 기록 문서 작성 시 "이걸 누가 읽을까?"를 먼저 확정 → 이원이면 분리 전제
- 파일명 컨벤션: `<topic>-pm.md` / `<topic>-dev.md`
- MEMORY.md 포인터엔 두 슬러그 함께 표기
- 관련 메모리: `feedback_ai_prose_patterns.md`
