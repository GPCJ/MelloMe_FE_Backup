## 1. API 명세 — Swagger 단일 소스 확정

**Q: 구글 시트에서 Swagger로 전환할 때 팀 내 반발은 없었나요?**
- 백엔드 개발자가 Spring Boot 기반이라 Swagger 자동 생성이 이미 되고 있었음 → 추가 부담 거의 없음
- 구글 시트에서 스펙 불일치가 반복되는 상황을 구체적 사례로 공유 → 자연스럽게 합의
- 핵심: "새 도구를 도입하자"가 아니라 "이미 있는 걸 공식으로 정하자"는 제안이었기 때문에 저항이 적었음

**Q: Swagger가 만능은 아닌데, 한계는 느꼈나요?**
- Swagger는 "현재 코드 상태"만 반영 → "앞으로 이렇게 바꿔달라"는 요청은 GitHub Issues로 별도 관리
- 응답 예시 데이터가 부족할 때가 있음 → MSW mock과 병행해서 프론트에서 자체 보완
- Swagger에 없는 암묵적 규칙(예: 페이지네이션 0-based)은 별도 문서화 필요

**Q: openapi-local.json을 로컬에 저장하는 루틴은 구체적으로 어떻게?**
- `curl -s http://EC2주소:8080/v3/api-docs > docs/openapi-local.json`
- 변경 감지: git diff로 이전 버전과 비교 → 프론트 영향 범위 파악
- CSV로도 API 목록을 관리해서 진행 상황 추적 (27개 API 순차 검토)

## 2. 백엔드 요청 — 전달 원칙

**Q: "LLM이 맥락을 파악할 수 있는 형태"란 구체적으로?**
- 백엔드 개발자가 이슈 텍스트를 AI에 복붙해서 해결하는 패턴을 발견
- 그래서 이슈 작성 시: 현재 동작 / 기대 동작 / 프론트 코드 스니펫 / 재현 스크린샷을 구조화해서 작성
- AI가 맥락을 잘못 해석하면 급방향 전환이 발생해서, "구현 방법"은 빼고 "문제와 기대 결과"만 명확히 기술

**Q: 이 원칙을 세우게 된 구체적 사건이 있나요?**
- 초기에 "이렇게 구현하면 될 것 같다"는 방식까지 이슈에 적었음
- 백엔드에서 AI 제안을 검토 없이 바로 적용 → 프론트 기대와 다른 방향으로 구현됨
- 수정 요청 → 또 AI가 다른 방향 제안 → 반복되는 급커브 경험
- 이후 "왜 문제인지"만 전달하고 구현은 위임하는 방식으로 전환 → 안정화

**Q: GitHub Issues를 두 레포에 동기화한다고 했는데, 왜 두 개?**
- 개인 백업 레포(MelloMe)와 팀 공식 레포(AIRO)가 분리되어 있음
- 백엔드 관련 이슈는 팀 레포에도 있어야 백엔드 개발자가 볼 수 있음
- 이슈 번호 매핑 테이블을 관리 (#2↔#1, #3↔#3 등)

## 3. 우선순위 분류 — P0/P1/P2

**Q: P0/P1/P2 기준은 어떻게 정했나요?**
- P0: "이게 없으면 데모 자체가 불가" — 핵심 플로우(회원가입→인증→글작성→필터→리액션/댓글)가 끊기는 항목
- P1: "MVP 필수지만 데모 시 '준비 중'으로 넘길 수 있는 것" — 팔로우, 블러 등
- P2: "MVP 범위이지만 당장 없어도 시연 가능" — 회원가입 토큰 반환 등
- 기준선: "Vercel 배포 URL 하나로 핵심 플로우를 보여줄 수 있는가"

**Q: 상태 추적(✅/❌/❓)은 얼마나 자주 업데이트했나요?**
- Swagger 스펙이 변경될 때마다 확인 → 해소된 항목 즉시 ✅ 처리
- 주간 회의 전에 전체 목록 리뷰 → 잔존 이슈 정리
- 처음 P0 3개 중 2개가 Swagger 확인으로 ✅ 해소 → 실제 남은 P0는 1개(title optional)

**Q: 우선순위를 백엔드에 어떻게 공유했나요?**
- GitHub Issues에 각 항목을 이슈로 등록 + 라벨로 우선순위 표시
- "P0 3개만 해결되면 시연 가능"이라는 한 문장이 가장 효과적이었음
- 전체 목록을 한번에 던지지 않고, P0부터 순차적으로 요청

## 4. 정책 변경 추적 — 연쇄 영향 범위 문서화

**Q: "기존 vs 변경" 형태로 문서화한다는 게 구체적으로?**
- 예: 인증 정책
  - 기존: 제출 → PENDING → 관리자 승인 → THERAPIST
  - 변경: 제출 → 즉시 THERAPIST + UNDER_REVIEW (사후 검토)
- 상태값도 명시: PENDING 폐기 → NOT_REQUESTED / UNDER_REVIEW / APPROVED / REJECTED 신규

**Q: 연쇄 영향을 어떻게 파악했나요?**
- 정책 변경 하나가 영향을 준 범위:
  1. 프론트 상태값 매핑 전면 수정
  2. 온보딩 플로우 변경 (가입 → 환영 → 인증 안내)
  3. 닉네임 입력 제거 → 랜덤 자동 부여 → 백엔드에 임시값 전송하는 workaround 코드
  4. title 필드 제거 → 빈 문자열 전송하는 workaround 코드
  5. 관리자 인증 검토가 Post-MVP → MVP P0으로 격상
- 이 전부를 한 문서에 모아두니, 나중에 "왜 이 임시 코드가 있지?" 할 때 바로 맥락 파악 가능

**Q: 임시 코드(workaround)를 어떻게 관리했나요?**
- 각 workaround에 "백엔드 OO 변경 시 제거 필요" 주석 or 별도 문서에 잔존 이슈로 기록
- 우선순위 문서의 ❌ 항목과 연결 → 백엔드 수정 완료 시 함께 정리

## 5. 기능명세 체계

**Q: 도메인별 순차 작성 방식을 왜 선택했나요?**
- 전체 기능을 한번에 명세하면 → 도메인 간 의존성 파악이 어렵고, 완성도가 고르지 않음
- 한 도메인을 끝까지 작성하면 → 해당 도메인의 정책/예외를 깊이 파악할 수 있음
- 실제로 인증 도메인을 먼저 완료(FNC-001~009)하면서 치료사 인증 정책 변경을 빠르게 반영 가능했음

**Q: 프론트/백엔드 독립 번호 체계는 왜?**
- 같은 기능이라도 프론트/백엔드 구현 범위가 다름 (예: 프론트에서 UI+상태관리, 백엔드에서 API+DB)
- 하나의 번호로 묶으면 진행률 추적이 모호해짐
- 각자 독립 번호로 관리하면서, 관련 항목은 도메인명으로 연결

**Q: CSV 내보내기는 왜 필요했나요?**
- 팀이 Google Sheets를 공유 문서로 사용하던 관행이 있었음
- 마크다운 명세와 Sheets를 이중 관리하지 않기 위해, CSV로 export해서 한 방향으로 동기화
- 명세 원본은 마크다운, Sheets는 뷰어 역할

## 6. 코드 수정 프로세스

**Q: grep 선행이 tsc보다 중요한 이유는?**
- tsc는 타입 에러만 잡음 → 런타임 의존성(예: 함수명을 문자열로 참조하는 경우)은 잡지 못함
- grep은 문자열 기반이라 모든 참조를 찾을 수 있음
- 실제 사례: setAuth 시그니처 변경 시 5개 파일 누락 → tsc는 일부만 잡았고, grep으로 전수 확인 후에야 완전히 해결

**Q: 이 프로세스를 팀 전체에 적용했나요?**
- 현재는 개인 프로세스 → 하지만 문서화해둬서 팀원이 참고 가능
- 핵심은 "프로세스의 존재"보다 "실수 → 원인 분석 → 재발 방지 루틴 수립"이라는 사이클 자체

## 7. 폴더 구조 리팩토링

**Q: flat에서 도메인별로 바꾼 기준은?**
- 컴포넌트가 15개 이상으로 늘어나면서, "이 컴포넌트 어디 있지?" 검색 빈도 증가
- auth, post, profile 등 도메인이 명확히 구분되는 시점에 리팩토링
- 변경 후: 도메인 폴더 하나만 열면 관련 컴포넌트가 전부 보임

**Q: import 경로가 전부 바뀌는데 리스크 관리는?**
- 변경 전: 모든 import를 grep으로 전수 확인
- 변경 후: `tsc -b` 빌드 검증으로 에러 0 확인
- shadcn/ui 경로(`@/components/ui/*`)는 의도적으로 변경하지 않음 → 외부 라이브러리 경로는 안정성 우선

**Q: 새 팀원이 왔을 때 이 구조를 이해할 수 있게 한 것은?**
- 변경된 트리 구조 + "새 파일은 해당 도메인 하위에 배치" 규칙을 문서에 명시
- 예시: 게시글 관련 → `components/post/`, `pages/post/`
- 규칙이 단순해서 별도 온보딩 없이도 따를 수 있음
