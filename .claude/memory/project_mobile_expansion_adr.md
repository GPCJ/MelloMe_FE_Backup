## ADR: 모바일 앱 확장 의사결정

**날짜:** 2026-03-26
**상태:** 부분 확정 (기술 최종 선택은 유저 테스트 후 결정)
**노션 업로드:** 완료 (2026-03-26) — 빌더스 리그 "🏗 설계 결정 & 아키텍처" 페이지

---

### 배경

멜로미는 발달장애 아동 치료사 커뮤니티 플랫폼으로, 현재 웹 MVP를 개발 중이다. 서비스 특성상 치료사들이 모바일 환경에서도 접근할 가능성이 높아, 모바일 앱 확장 여부를 사전에 검토할 필요가 있었다.

### 검토한 선택지

1. **PWA (Progressive Web App)**
   - 기존 웹 코드를 거의 그대로 사용하면서 홈 화면 아이콘, 오프라인 캐싱 등을 추가하는 방식
   - 추가 개발량이 가장 적고(1~2일), 별도 코드베이스 불필요
   - 단, 앱스토어 배포 불가, 일부 네이티브 기능(푸시 알림 등) 제한

2. **Capacitor (하이브리드 앱)**
   - 기존 웹 앱을 네이티브 껍데기(WebView)로 감싸서 앱스토어에 배포하는 방식
   - 기존 React + Tailwind + shadcn 코드 약 90% 재사용 가능
   - 앱스토어 배포 가능하고, 플러그인을 통해 푸시 알림 등 네이티브 기능 연결 가능
   - 추가 개발량 1~2주 수준

3. **React Native (네이티브 앱)** → **제외됨**
   - React 문법으로 네이티브 앱을 처음부터 새로 만드는 방식
   - 성능과 사용자 경험이 가장 좋지만, 기존 UI 코드 재사용 불가
   - 프론트엔드 개발자 1명 현실에서 감당 불가 → 제외

### 논의 결과

1. **타이밍** → MVP는 웹으로 진행
2. **수요** → 유저 테스트로 검증 예정 (추측 아닌 데이터 기반 결정)
3. **기술 선택** → React Native 제외, PWA 또는 Capacitor로 좁힘
4. **앱스토어 배포** → 미정 (기술 확정 후 논의)

### 다음 단계

1. 웹 MVP 완성 및 배포
2. 유저 테스트 진행 → 모바일 수요 데이터 수집
3. 수요 확인 시 → PWA vs Capacitor 최종 결정
4. 앱스토어 배포 필요 여부에 따라 기술 확정

### 이 결정에서 배운 점

- 기술 선택은 "무엇이 가장 좋은가"가 아니라 "현재 팀이 감당할 수 있는가"가 기준이 되어야 한다
- 사용자 수요가 검증되지 않은 상태에서 기술 투자를 결정하는 것은 위험하다
