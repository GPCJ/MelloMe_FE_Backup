UI 위치, 레이아웃, 컴포넌트 디자인은 디자이너 확인 없이 임의로 구현하지 말 것.

**Why:** 협업 프로젝트에서 프론트엔드가 디자이너 공유 없이 UI를 구현하면 나중에 재작업이 발생함. 첨부파일 UI 위치 결정 시 이 이슈가 발생함 (2026-03-30).

**How to apply:** UI 구현 요청 시 디자인/위치가 확정되지 않은 경우 타입/로직만 먼저 구현하고 UI는 보류. 디자이너 확인 후 UI 작업 진행. **브레인스토밍 단계에서도 동일** — 알림 드롭다운/별도 페이지 구성 등 UI 레이아웃 질문은 디자이너 확정 전이면 Skip하고 아키텍처/타입/로직 설계에 집중.

## 백엔드 API 변경이 디자인에 영향 줄 때 (2026-04-21 추가)

백엔드 API 변경사항이 라벨/아이콘/색상/표시 조건 등 디자인 결정에 영향을 주는 경우, 백엔드 명세대로 바로 UI 구현하지 말 것. **개발 용어 뺀 질문 초안**을 만들어 디자이너에게 먼저 의도 확인.

**Why:** 백엔드 스펙만 따르면 디자이너의 UX 의도가 반영되지 않을 수 있음. 라벨/아이콘/색상/카운트 노출 정책 같은 결정은 디자이너 몫. 사용자 표명: "디자인적인 부분에 변경이 있어서 이런 부분은 내가 백엔드 명세대로 사용해서 UI를 개발하는 것이 아닌 디자이너의 의도를 물어야한다고 생각해. 그렇기 때문에 개발 용어를 최대한 쉽게 바꾸던가 빼서 디자이너에게 이런 부분이 이번에 추가, 수정됐는데 어떻냐고 물어야해" (2026-04-21).

**How to apply:**
1. 백엔드 변경 공유 받으면 **디자인 영향 지점 식별** (라벨, 아이콘 의미, 색상 토큰, 카운트 표시 정책 등)
2. "enum/필드/응답/payload" 같은 개발 용어 빼고 **"사용자가 보게 되는 부분"**으로 질문 초안 작성
3. 디자이너 답변 후 코드 작업 시작
4. 디자이너가 "일단 임시로 유지하자"고 하면 코드에 **임시 유지 주석 + 컨펌 대기 항목**을 남김

**근거 사례:** 리액션 API 리네임 대응 시 공감/잘봤어요/유익 → 좋아요/궁금해요/유용해요 변경을 바로 적용하지 않고 디자이너에게 의도 확인 → "아이콘만 표시할 거라 라벨은 무관, 토큰명은 임시 유지, 카운트 0이면 아이콘만" 결정 받아냄. (`project_post_reaction_api_rename.md`)
