## 아키텍처 결정: SSE 알림 기능 (2026-04-02)

### 0단계: 실시간 기술 선택 — Polling / WebSocket / SSE 중 SSE

| 방식 | 설명 | 멜로미 적합성 |
|------|------|--------------|
| Polling | 클라이언트가 주기적으로 "새 알림 있어?" 반복 요청 | 불필요한 요청 낭비, 실시간성 낮음 |
| WebSocket | 양방향 실시간 통신 (채팅 등) | 알림은 서버→클라이언트 단방향이라 과함 |
| **SSE** | 서버→클라이언트 단방향 스트리밍 | 알림 용도에 딱 맞음, 가볍고 HTTP 기반 |

백엔드가 SSE로 구현 중이기도 하고, 멜로미 알림은 단방향(서버가 push)이라 SSE가 적합.

### 1단계 결정: 옵션 B — Zustand store 안에 SSE 연결 통합

### 검토한 3가지 옵션

**옵션 A: useSSE 커스텀 훅 + Zustand store 분리**
- SSE 연결 로직은 `useSSE` 훅에, 알림 상태는 Zustand store에 분리
- 장점: 관심사 분리가 명확
- 단점: 훅이 store를 구독하고 store가 훅의 이벤트를 받아야 해서 연결이 번거로움. 컴포넌트 언마운트 시 연결 끊김 위험

**옵션 B: Zustand store 안에 SSE 연결 통합 (채택)**
- store가 `connect()` / `disconnect()` 메서드를 갖고 SSE 연결과 알림 상태를 한 곳에서 관리
- 장점: 연결 생명주기와 상태가 한 곳에 응집, 컴포넌트 마운트/언마운트와 무관하게 연결 유지 가능, 멜로미 규모에 적합
- 단점: store가 비대해질 수 있으나 MVP에서는 문제 안 됨

**옵션 C: NotificationService 싱글톤 클래스 + Zustand**
- 별도 서비스 클래스가 SSE 연결 관리, Zustand는 순수 상태만
- 장점: 테스트 용이, 확장성 좋음
- 단점: MVP 규모에 비해 과한 추상화, 클래스 기반이라 React 생태계와 이질적

### 선택 근거
1. **응집도**: SSE 연결과 알림 상태가 밀접하게 결합 → 한 store에서 관리하는 게 자연스러움
2. **생명주기 독립성**: Zustand store는 컴포넌트 밖에서도 유지 → 페이지 이동해도 연결 안 끊김
3. **MVP 적합성**: 알림 이벤트 5종(댓글/대댓글/팔로우/리액션/공지) 규모에 옵션 C는 과도
4. **기존 패턴 일관성**: 멜로미 프로젝트가 이미 Zustand store 중심으로 상태 관리 중

### 2단계 결정: 토큰 인증 방식 — C안 (fetch-event-source 라이브러리)

**배경**: 브라우저 기본 `EventSource` API는 커스텀 헤더를 지원하지 않아 Authorization 헤더에 Access Token을 넣을 수 없음

| 방법 | 설명 | 판단 |
|------|------|------|
| A. 쿼리 파라미터 | `/sse?token=xxx` | 토큰이 URL에 노출 → 보안 우려 |
| B. fetch 직접 파싱 | fetch + ReadableStream 수동 처리 | 가능하지만 재연결/에러처리 직접 구현 부담 |
| **C. `@microsoft/fetch-event-source`** | 내부적으로 B를 해주며 재연결/에러 핸들링 내장 | **채택** |

선택 근거: 기존 Authorization 헤더 방식 그대로 유지, 백엔드 추가 수정 불필요, 보안 우려 없음, 구현 부담 최소
