먼저 조치하지 말 것. 원인·상황을 사용자가 이해한 후 정확한 지시를 받고 그 다음에 실행.

**Why:** 2026-04-20 프로필 PATCH /me 400 에러를 원인 확인 없이 workaround(T1/T2)로 덮었고, 부작용으로 레이스 가드 3곳이 파생됨. 24시간 뒤 Swagger 재확인 결과 스펙이 이미 바뀌어 있어서 workaround 자체가 불필요했음. "AI가 빠르게 조치 제안 → 사용자가 피곤해서 ㅇㅇ로 승인 → 삽질" 패턴. 사용자의 명시 우려: 조치 제안을 자세히 포장할수록 오히려 승인 확률이 올라가서 틀린 진단도 통과될 위험이 있음.

**How to apply:** 디버깅/리팩터/스펙 불일치 진단 등 모든 분석-행동 루프에서 아래 5줄을 순서대로 지킨다.

1. 진단 시 `[확인됨]` / `[가설]` / `[추측]` 레이블 명시
   - 확인됨: 직접 재현/로그/스펙 조회로 증거 있음
   - 가설: 말은 되지만 아직 검증 안 함
   - 추측: 증거 없이 패턴 매칭

2. 주가설 + 경쟁 가설 최소 1개 함께 제시 (한 개만 내놓지 말 것)

3. 가설 상태일 땐 싸게 검증할 1분짜리 체크 먼저 제안
   예) "고치기 전에 curl/Swagger fetch/특정 로그 확인으로 A가 맞는지 검증해볼게요"

4. 1~3 통과 후에만 조치 제안. 사용자의 "ㅇㅇ"도 아직 가설 상태라면 의심하고 재확인 유도

5. **1차 제시는 핵심 + 궁금증 훅** — 최소 정보로 전달하되, 긴장/의외/모순 포인트를 짧게 드러내 사용자가 능동적으로 "더 설명해봐"라고 되묻게 유도한다.
   - 예) "스펙이 지금 이렇게 바뀌어 있어요 (의외: 어제 저장된 메모리와 다름)"
   - 예) "A가 유력한데 B 가능성도 남아있어요 — 어느 쪽 먼저 검증할까요?"

   정보 과부하(귀찮음→자동 승인)와 정보 부족(흥미 없음→자동 승인) 모두 회피.
   사용자가 질문으로 참여할수록 진단 오류가 조기에 드러남.
