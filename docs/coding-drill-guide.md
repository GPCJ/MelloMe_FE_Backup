# 코딩 드릴 가이드 (v2 — 설계 설명 연습)

> 목표: 코드 **타이핑 능력**이 아니라 **설계 언어화 능력**을 키운다.
> 면접/PR 리뷰에서 "왜 이렇게 설계했나"를 1~2분 안에 말할 수 있게 만드는 것이 목적.

---

## 왜 v2인가

v1(백지 재구현)은 완료 — 프론트 전체 코드를 이미 한 번 훑었고,
실전 트러블슈팅도 AI 없이 주도하고 있음.

남은 약점은 **"이 구조를 왜 이렇게 잡았는가"를 말로 설명하는 능력**.
v2는 여기에 집중.

---

## 루틴 구조 (주 2~3회 × 30분)

| 단계 | 시간 | 내용 |
|------|------|------|
| 1. 대상 선정 | 2분 | 최근 1주 내 작업한 파일 **1개** (hot paths 우선: PostEditPage, useFileAttachment 등) |
| 2. 자문 | 10분 | 아래 질문 풀에서 **2~3개** 골라 답변 구상 (머릿속 또는 키워드 메모) |
| 3. 답변 작성 | 15분 | 면접 답변 톤으로 서술. **말하듯이** 쓴다. 완벽한 문장 아니어도 OK |
| 4. 기록 | 3분 | `coding-drill-log.md`에 "질문 / 답변 요점 / 막힌 부분" 3줄 |

---

## 질문 풀

대상 파일을 정했으면 아래 중 2~3개를 골라 답변한다.

### 경계 / 책임
- 이 컴포넌트를 왜 분리했나? 어디까지가 이 컴포넌트의 책임인가?
- 이 훅을 왜 별도 훅으로 뽑았나? 컴포넌트 안에 두면 안 되는 이유?
- 이 유틸/API 함수는 왜 이 파일에 있나?

### 상태 / 데이터 흐름
- 이 상태가 왜 **여기**(컴포넌트 local / 훅 / 스토어)에 있나?
- props vs context vs store 중 왜 이것을 골랐나?
- 상태 변경 트리거는 어디서 오고, 어디까지 전파되나?

### 의존 방향 / 구조
- 이 파일은 누구에게 의존하고, 누구에게 의존받나?
- 폴더 경계(`components/`, `pages/`, `hooks/`, `stores/`)를 왜 이렇게 나눴나?
- 백엔드 API 변경 시 어느 파일부터 건드려야 하나?

### 면접 모의
- 이 패턴을 면접관에게 **1분**에 설명한다면 어떻게?
- "더 좋은 방법 있었는데 왜 이걸 골랐나?" 질문에 답할 수 있나?
- 이 코드의 트레이드오프는 무엇인가?

---

## 규칙

1. **AI 답변 금지** — 자문 및 답변 작성 단계에서 AI에게 묻지 않는다. 다 쓰고 나서 AI와 비교는 OK.
2. **완벽하지 않아도 OK** — 답이 흐릿하면 그게 기록할 약점.
3. **주 2~3회** — 매일 부담 버림. 실제 작업한 파일이 재료라 자연스럽게 쌓임.
4. **주간 노션 업로드** — `/update-drill` 커맨드로 주간 배치 업로드.

---

## 평가 루프

- **2주 후** 면접 모의질문으로 체감 효과 검증
- 효과 약하면 질문 풀 재조정, 시간/빈도 재검토
- 효과 있으면 질문 풀 확장 (성능, 보안, 접근성 등 추가)
